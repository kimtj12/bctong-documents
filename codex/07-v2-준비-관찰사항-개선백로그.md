# 07. v2 준비 관찰사항 및 개선 백로그

## 1) 한 줄 요약
현재 시스템은 업무가 풍부하고 실무형 로직이 많이 축적되어 있으나, 상태 전이·권한·자동화·연동 안정성 관점에서 구조 정리가 필요합니다.

## 2) 우선순위별 개선 백로그

## P0 (v2 착수 초기에 반드시)
1. 상태머신 정식화
- 일반오더/선착불오더 상태 전이를 단일 상태도와 이벤트 사전으로 명문화
- 허용 전이/금지 전이/롤백 전이를 API 레벨에서 일관 강제

2. 비밀정보/연동키 보안화
- 코드 내 하드코딩된 API 키/시크릿/웹훅을 즉시 환경변수/시크릿 매니저로 이전
- 키 회전 정책/권한 최소화 정책 수립

3. 권한 모델 재설계
- role 기반 + 데이터 스코프 기반 정책을 표준화
- 특정 사용자 ID 하드코딩 권한 제거(회계 특수권한 정책화)

4. 배치/웹훅 관측성 확보
- 자동매칭/콜동기화/웹훅 처리 성공률·실패사유 대시보드화
- 장애 시 재처리 큐/수동복구 절차 표준화

## P1 (운영 생산성/정확도 향상)
1. 일반/선착불 오더 모델 통합 검토
- 중복 화면/중복 로직 감소
- 수수료 정책을 상태 분기 대신 속성 정책으로 전환 검토

2. 청구/수금 회계 용어 통일
- `price`, `taxPrice`, `income`의 UI 라벨과 보고서 기준 통일
- "표준청구 vs 실발행" 차이를 명확하게 표시

3. 이벤트 로그 체계 강화
- Notification 중심 로그를 감사로그(Audit) 모델로 승격
- 누가/언제/왜/무엇을 바꿨는지 표준 필드화

4. 데이터 품질 검증 룰 도입
- 금액 음수/비정상 연락처/필수값 누락 방지
- 주문-청구-수금-정산 간 정합성 검증 작업 자동화

## P2 (확장/분석 고도화)
1. 분석 계층 분리
- 운영 DB 조회 기반 분석을 집계 테이블 또는 DWH로 분리
- 메트릭 정의서(건수/매출/수익률/업체수) 공식 문서화

2. 정책 엔진화
- 지급예정일 계산, 수수료/공제 로직, 결제예정일 계산을 설정형으로 이전

3. 대외 연동 표준화
- 24시콜/팝빌/알림톡 어댑터 계층 분리
- 연동별 SLA, 재시도, 서킷브레이커 전략 적용

## 3) 기능 단위 리스크 메모
- 주문 생성/수정: 풍부하지만 입력 검증이 화면 중심이라 API 표준검증 강화 필요
- 매입 자동매칭: 실무 효율 높지만 실패 큐 운영정책(처리기한/담당자/재시도) 명확화 필요
- 청구 발행: 팝빌/수기 공존 구조는 현실적이나, 발행취소/재발행 시나리오 명확화 필요
- 수금 관리: 분할입금 지원은 좋으나 입금자명/증빙 연계 표준이 약함

## 4) v2 설계 원칙 제안
1. 상태는 한 곳에서만 바꾼다(상태 전이 서비스)
2. 금액은 정의를 분리한다(청구금액/발행금액/입금금액/지급금액)
3. 권한은 역할+범위+행위 3축으로 관리한다
4. 자동화는 관측 가능해야 한다(실패가 보여야 운영 가능)
5. 외부연동은 실패를 기본값으로 설계한다

## 5) 마이그레이션 전략 제안
1. 현행 동작 보존을 우선하는 "호환 레이어" 설계
2. 주문/정산/청구를 순차적으로 모듈화
3. 기존 데이터의 상태/금액 정합성 사전 점검 후 이관
4. 관리자 화면부터 점진 교체(운영 영향 최소화)
