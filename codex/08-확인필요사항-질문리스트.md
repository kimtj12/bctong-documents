# 08. 확인 필요사항 및 질문 리스트

본 항목은 코드만으로 확정하기 어려운 정책/운영 사실을 정리한 체크리스트입니다.

## A. 비즈니스 정책 확인
1. 일반오더와 선착불오더를 v2에서도 분리 상태로 유지할지, 단일 모델로 통합할지?
2. `pending` 상태는 실운영에서 실제 사용되는지, 과거 호환값인지?
3. 포인트 적립 "배차 완료 후 15일 이내" 정책이 현재 정확히 어디에서 보장되는지(코드/운영수동)?
4. 즉시지급(`driverQuickPay`) 사용 기준과 승인 절차가 무엇인지?
5. 미수 기준일(`paymentDueDate`) 초과 시 후속 조치(독촉/중단/승인)가 정의되어 있는지?

## B. 권한/조직 정책 확인
1. 회계 고급권한(특정 ID 하드코딩) 정책의 공식 기준은 무엇인지?
2. `partner`와 `freelancer`의 최종 업무 경계(수정 가능 범위, 승인 권한)는 어디까지인지?
3. 사용자 계정의 `ceo` 권한은 현재 실제로 쓰는지(일부 코드에서는 표시만 존재)?

## C. 데이터/기능 정합성 확인
1. 웹에서 사용하는 `/payment-logs`, `/hubs`는 현 API 코드에 정의가 보이지 않음.
- 별도 서버/레거시 API를 쓰는지?
- 아니면 미사용 잔존 코드인지?
2. 주석 처리된 메뉴/기능(예: 일부 실험실 항목)은 폐기인지, 재활성 예정인지?
3. 선착불 완료 후 정산/지급 상태(`withdraw/finished`)와의 연결 정책이 운영상 어떻게 정의되는지?

## D. 연동/운영 확인
1. 24시콜 장애 시 표준 복구 절차(수동 배차 전환, 데이터 동기화 재시도)가 문서화되어 있는지?
2. 팝빌 발행 실패 시 재발행/수기전환 기준이 있는지?
3. 유니패스 조회값(장치장 연락처 등)을 실제 업무 의사결정에 어떻게 활용하는지?

## E. v2 기획을 위한 결정 필요 항목
1. 상태 체계를 업무 단위 중심으로 재정의할지(예: 배차/정산/청구를 별도 서브상태로 분리)?
2. 운영 로그를 Notification에서 Audit Log로 격상할지?
3. 청구/수금 KPI 기준(실발행총액 vs 표준청구금액)을 무엇으로 단일화할지?
4. 분석 지표 공식(수익률 계산 방식)을 조직 표준으로 확정할지?

## F. 권장 인터뷰 대상
- 배차 운영자(일반/선착불 각각)
- 회계/정산 담당자
- 세금계산서 발행 담당자
- 영업 프리랜서 대표 사용자
