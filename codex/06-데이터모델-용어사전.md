# 06. 데이터 모델 용어사전

본 문서는 개발 구현이 아니라 "업무 데이터 의미" 중심으로 정리했습니다.

## 1) 핵심 엔티티 요약

| 엔티티 | 업무 의미 | 핵심 관계 |
|---|---|---|
| Order(주문) | 배차 1건의 전체 생명주기 | Company, User, Manager, IncomingTax, PointLog |
| Company(회사) | 화주사 및 정산 정책 단위 | Users, Orders, Partners, Items, Prices, Invoices |
| Partner(거래처) | 상/하차지 주소/연락처 마스터 | Company |
| Item(상차품) | 화물 품목/이미지 마스터 | Company |
| Incoming Tax(매입세금계산서) | 기사 정산 근거 문서 | Order(1:1) |
| Invoice(청구) | 화주 청구 단위(월별/수기 포함) | Company, InvoiceLogs, OrderList |
| Invoice Log(수금로그) | 입금 분할 기록 | Invoice |
| Point Log(포인트로그) | 적립/차감 이력 | User, Order |
| Notification(알림) | 운영 변경 이력 | relationType/relationId로 대상 연결 |
| Driver(기사관리) | 블랙/화이트리스트 | registerUserId(등록자), orderId 참조값 |
| Price(단가표) | 회사별 구간/톤수 단가 | Company |
| Distance Price(거리단가) | 거리x차종 평균매입 기준표 | 분석용 집계 데이터 |
| Giftcard(상품권) | 발송 템플릿/설정 | 운영설정 |

## 2) Order(주문) 주요 필드

### 2-1. 식별/관계
- `id`
- `company` 화주사
- `user` 요청자(등록자)
- `manager` 배차 담당자
- `incoming_tax` 연결된 매입 계산서
- `point_log` 연결 포인트로그

### 2-2. 상태/일정
- `status`: `pending/requested/accounting/withdraw/finished/canceled/feeRequested/feeFinished/feeCanceled`
- `startType/endType`: `today/tomorrow/monday/etc`
- `startDate/endDate`, `loadTime/unloadTime`, `driverTime`
- `matchDatetime`, `payReserveDate`, `driverPayDate`, `driverFinalDate`

### 2-3. 장소/연락
- `startName/startAddress/startAddressDetail/startLat/startLng/startPhoneNumber`
- `endName/endAddress/endAddressDetail/endLat/endLng/endPhoneNumber`

### 2-4. 운송/옵션
- `carWeight/carType`
- `isReceiptRequired/isRoundTrip/isMixed/mixLength`
- `productWeight/productContent`, `memo`
- `unipassCode`

### 2-5. 기사/정산
- `driverName/driverPhoneNumber/driverCarNumber/driverCarInfo`
- `driverPrice/price/fee`
- `driverAddedPrice/ownerAddedPrice` 및 사유
- `isExcludeTax`, `driverQuickPay`
- `driverBusinessNumber/driverCompayName/driverBank/driverBankName/driverBankNumber`

### 2-6. 연동/첨부
- `callOrderId/callOrderStatus/callOrderBody/callOrderUpdateDate/callOrderManagerName`
- `productImages/orderImages/receiptImage/do/ic/pl/umsFiles`
- `distance/time/fare`
- `billMonth`

## 3) Company(회사) 주요 필드

### 3-1. 기본 정보
- `name`, `businessNumber`, `ceoName`
- `address/lat/lng`
- `phoneNumber/mobileNumber/faxNumber/email`

### 3-2. 세금계산서/결제 정책
- `taxName/taxCeoName/taxBizUptae/taxBizJongmok`
- `taxManagerName/taxManagerPhoneNumber/taxEmail` (+ 보조 담당자)
- `invoiceMonth`, `invoiceDate` (결제예정일 계산 기준)

### 3-3. 운영 정책/문서
- `isHideSmsPrice` (SMS 금액 노출 여부)
- `contractMemo`, `internalMemo`, `loadTimeMemo/endTimeMemo/lunchTimeMemo`
- `businessFile/contractFile/insuranceFile/internalImages`
- `priceTableGoogleSheetUrl/originalGoogleSheetUrl`

### 3-4. 조직 관계
- `manager` 정담당
- `partner` 부담당
- `users`, `orders`, `invoices`, `partners`, `items`, `prices`

## 4) Invoice(청구) 주요 필드
- `company`, `month`, `orderList`
- `price` 표준청구공급가액
- `taxPrice` 실발행총액(공급가+부가세)
- `income`, `incomeDate`, `isPaid`
- `mgtKey`, `ntsCode` (발행 식별)
- `taxBusinessNumber/taxCompanyName/taxDetail/taxMemo`
- `paymentDueDate`
- `isManual`, `manualMemo`, `manualFile`
- `invoice_logs`

## 5) Incoming Tax(매입 계산서) 주요 필드
- `issueDatetime`
- `businessNumber/companyName/companyCeoName`
- `amount`
- `carNumber`
- `info` (은행/계좌/예금주 텍스트 단서)
- `ntsConfirmNumber`
- `raw` (원본 payload)
- `status`: `pending/approved/declined/canceled/automatch/review`
- `failReason`: `NO_ORDER/MULTIPLE_ORDERS/TAX_ISSUE/NO_BANK_INFO`
- `declineMemo`
- `order`

## 6) 포인트/알림/기사 엔티티

### 6-1. Point Log
- `amount` (+/-)
- `afterAmount` (반영 후 잔액)
- `reason`
- `user`, `order`
- `gift` (확장 필드)

### 6-2. Notification
- `name`, `type`, `content`
- `requestUser`, `requestCompany`
- `relationType`, `relationId`
- `isDone`

### 6-3. Driver
- `type`: `blacklist/whitelist`
- `phoneNumber`, `name`, `carNumber`, `carInfo`
- `reason`, `orderId`, `registerUserId`

## 7) 사용자(User) 확장 필드
- `permission`: `user/freelancer/partner/admin`
- `company` (소속회사)
- `points`
- `managedOrders/managedCompanies/partnerCompanies`
- `ddID`, `color`

## 8) 데이터 품질/운영 관점 메모
- 동일 비즈니스 개념이 화면마다 다른 표현으로 보일 수 있음(예: 배차완료/계산서)
- 상태값은 강한 업무 의미를 가지므로, v2에서 상태 정의 문서를 단일화할 필요가 큼
- 금액 필드(`price`, `driverPrice`, `taxPrice`, `income`)의 의미 차이를 UI에서 더 명확히 구분할 필요가 있음
