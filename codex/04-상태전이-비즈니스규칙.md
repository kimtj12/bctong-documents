# 04. 상태전이 및 비즈니스 규칙

## 1) 주문 상태 체계

### 1-1. 일반 오더 상태
- `pending` 대기
- `requested` 배차중
- `accounting` 배차완료(기사/운임 확정)
- `withdraw` 계산서 접수/지급대기
- `finished` 지급완료
- `canceled` 취소

### 1-2. 선착불 오더 상태
- `feeRequested` 배차중
- `feeFinished` 완료
- `feeCanceled` 취소

## 2) 상태전이 트리거

| 전이 | 트리거 | 주요 조건/행동 |
|---|---|---|
| `requested -> accounting` | 배차상황실 기사정보 저장 | 기사정보/운임 입력 완료 |
| `accounting -> withdraw` | 정산정보 입력(수기/자동매칭/수동매칭) | 계좌/사업자 정보, 지급예정일 입력 |
| `withdraw -> finished` | 지급 처리(단건/일괄) | 지급일(`driverFinalDate`) 기록 |
| `requested -> canceled` | 화물 취소 | 주문 취소 알림 생성 |
| `accounting -> requested` | 배차취소(매칭 취소) | 기사정보/운임값 제거 |
| `feeRequested -> feeFinished` | 선착불 배차 확정 | `fee` 필수, `price=driverPrice+fee` |
| `feeFinished -> feeRequested` | 선착불 배차 취소 | fee/price 초기화 |
| `* -> finished` | 배치 완료(`/batch-finish`) | 선택 주문 일괄 완료 |

## 3) 날짜 규칙

### 3-1. 요청일 규칙
- `today/tomorrow/monday/etc`를 실제 날짜로 환산하여 저장
- today 하차는 `unloadTime = 바로당착`

### 3-2. 차주 지급예정일 계산
- 계산식
  - 기준일(계산서 발행일)의 요일을 Excel 방식(월=1, 일=7)로 변환
  - `payReserveDate = 기준일 + (30 - weekdayExcel)일`
- 화면/크론 로직 모두 동일한 공식을 사용

### 3-3. 화주 결제예정일 계산
회사별 설정값 사용
- `invoiceDate = 0`이면 작성일 + 1일
- `invoiceDate = 100`이면 `(작성일 + invoiceMonth개월)의 말일`
- 그 외 값이면 `(작성일 + invoiceMonth개월)의 해당 일자(invoiceDate)`

## 4) 금액/정산 계산 규칙

### 4-1. 산재 공제
- `driverSanjaeFee = floor((driverPrice * 0.501 * 0.0088) / 10) * 10`

### 4-2. 즉시지급 수수료
- `quickPayFee = driverPrice * 1.1 * 0.015`

### 4-3. 최종 지급액
- 기본: `driverPrice*1.1 - 산재공제`
- 산재 제외: `driverPrice*1.1`
- 즉시지급: 기본식에서 즉시지급 수수료 추가 차감

## 5) 매입세금계산서 자동매칭 규칙

### 5-1. 기본 매칭 조건
- 주문 상태가 `accounting`
- 주문 `driverPrice == 계산서 amount`
- 주문 차량번호가 계산서 비고(remark) 단서와 부분 일치

### 5-2. 예외/실패 처리
- 매칭 0건: `NO_ORDER`
- 매칭 2건 이상: `MULTIPLE_ORDERS`
- 비고에 산재 이슈 포함: `TAX_ISSUE`
- 은행/예금주/계좌 파싱 실패: `NO_BANK_INFO`

### 5-3. 성공 후 후속처리
- 주문 `withdraw` 전환
- 주문에 매입세금계산서 연결
- 기사 계좌정보 업데이트
- 포인트 +1000(비관리자 주문)
- 기사 안내 알림톡 발송

## 6) 청구/수금 상태 판정 규칙

Invoice 관점 분류
- 청구전: `mgtKey is null`
- 입금전: `isPaid=false` 그리고 `paymentDueDate > today`
- 미수: `isPaid=false` 그리고 `paymentDueDate <= today`
- 완료: `isPaid=true`

수금 완료 판정
- `income == taxPrice`이면 `isPaid=true`

## 7) 운영 품질 규칙
- 가격/수수료 변경 시 알림 생성(변경 전후 금액 기록)
- 오더 수정/취소 시 알림 생성(추적 가능성 확보)
- 기사 연락처/차량정보는 매칭 해제 시 반드시 초기화

## 8) 화면 전용 상태표현(주의)
- 히스토리 화면에서 신규 접수 3분 이내 주문은 `접수(register)`로 표시
- 이는 저장 상태값이 아니라 화면 표시용 파생 상태
