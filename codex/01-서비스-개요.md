# 01. 서비스 개요

## 1) 배차통이 하는 일
배차통은 화주(고객사)의 운송 요청을 접수하고, 배차 완료 후 정산/청구/수금까지 연결하는 운영 시스템입니다.

핵심은 다음 5단계입니다.
1. 배차 요청 접수
2. 기사 매칭 및 운행 진행
3. 기사 정산(매입 계산서 기준)
4. 화주 청구(매출 계산서/명세)
5. 수금 및 미수 관리

## 2) 시스템 구성
- `web`: 운영 대시보드(배차/정산/청구/관리)
- `api`: Strapi 기반 API 서버(데이터/비즈니스 로직/외부 연동)

업무 관점으로 보면,
- `web`은 운영자가 의사결정을 내리고 실행하는 콘솔
- `api`는 상태 관리, 데이터 저장, 자동화, 외부 시스템 통신을 담당

## 3) 주요 사용자 그룹
- 업체(`user`): 본인 회사의 배차 요청/이력/거래명세/마일리지/거래처·상차품 관리
- 배차 매니저(`partner`): 배차 운영 참여, 일부 관리자성 메뉴 사용
- 영업 프리랜서(`freelancer`): 본인 담당 회사 범위 중심 조회/운영
- 관리자(`admin`): 전체 운영/정산/청구/계정/정책성 데이터 관리

## 4) 핵심 도메인(업무 객체)
- 주문(`Order`): 배차 단위의 중심 엔티티
- 회사(`Company`): 화주사/정산사 정책 단위
- 거래처(`Partner`): 상/하차지 마스터
- 상차품(`Item`): 화물 품목 마스터
- 매입세금계산서(`Incoming Tax`): 기사 정산 근거
- 청구(`Invoice`): 화주 대상 청구 단위
- 수금로그(`Invoice Log`): 분할/누적 입금 이력
- 포인트로그(`Point Log`): 사용자 마일리지 증감 이력
- 기사관리(`Driver`): 블랙/화이트리스트
- 알림(`Notification`): 가격변경/오더수정/취소 등 운영 이력

## 5) 현재 시스템의 특징
- 주문 상태가 업무 단계를 직접 표현함(배차중/배차완료/계산서/지급완료/취소 + 선착불 전용 상태)
- 매입 계산서 기반 자동 매칭이 존재하며, 성공 시 정산 단계로 자동 전환됨
- 청구/수금 파트는 "청구 생성 → 세금계산서 발행 → 입금 누적"의 분리 구조
- 운영자 수동 처리와 자동 처리(크론)가 혼재되어 있음

## 6) v2 기획에 중요한 현재 구조적 포인트
- 상태 전이가 화면/자동화/수동입력 경로로 분산됨
- 동일한 업무가 일반오더/선착불오더로 분기되어 중복 운영됨
- 외부 연동 장애 시 수동 복구 절차가 필요한 구간이 있음
- 일부 데이터/메뉴는 실제 운영 활용 여부 확인이 필요함
